# Stakeholder Communication Checklist

## Purpose
This checklist helps Project Managers maintain consistent, transparent communication with stakeholders throughout the project lifecycle. Use this template to ensure all stakeholder groups receive timely, relevant information.

## Pre-Project Phase

### Stakeholder Identification
- [ ] List all stakeholder groups (e.g., executives, product teams, engineering, sales, support, customers)
- [ ] Identify primary contact for each group
- [ ] Document decision-making authority for each stakeholder
- [ ] Determine communication preferences (email, Slack, meetings, reports)
- [ ] Define information needs for each stakeholder group

### Communication Plan Setup
- [ ] Create communication calendar with recurring touchpoints
- [ ] Establish escalation paths for urgent issues
- [ ] Define communication channels and protocols
- [ ] Set expectations for response times
- [ ] Document where project information will be maintained (single source of truth)

## During Project Execution

### Weekly Communications
- [ ] Send weekly status update to all stakeholders (see template in octoacme-risks-and-communication.md)
- [ ] Update project board or dashboard with latest progress
- [ ] Flag any risks or blockers requiring stakeholder attention
- [ ] Highlight upcoming milestones or decision points
- [ ] Share demo links or artifacts when available

### Milestone Communications
- [ ] Pre-milestone: Confirm scope, timeline, and success criteria
- [ ] At milestone: Share completion status and key outcomes
- [ ] Post-milestone: Distribute retrospective insights and next steps
- [ ] Update roadmap or timeline if adjustments are needed

### Change Communications
- [ ] Notify stakeholders immediately of scope changes
- [ ] Document rationale for timeline adjustments
- [ ] Communicate resource or priority shifts
- [ ] Obtain necessary approvals before proceeding
- [ ] Update project charter or one-pager to reflect changes

### Risk and Issue Communications
- [ ] Alert stakeholders to new high-impact risks within 24 hours
- [ ] Provide mitigation plan with each risk notification
- [ ] Escalate blockers that require stakeholder intervention
- [ ] Follow up when risks are resolved or mitigated
- [ ] Maintain risk transparency in all status updates

### Decision Communications
- [ ] Document decisions made and rationale
- [ ] Share decision log with affected stakeholders
- [ ] Confirm stakeholder approval for go/no-go decisions
- [ ] Update team on direction after decision meetings

## Release Phase

### Pre-Release
- [ ] Share release plan and timeline with all stakeholders
- [ ] Communicate testing results and readiness criteria
- [ ] Provide release notes draft for review
- [ ] Notify support and customer-facing teams of upcoming changes
- [ ] Schedule release announcement

### Release Day
- [ ] Announce deployment start and expected completion
- [ ] Provide real-time updates if issues arise
- [ ] Confirm successful deployment and verification
- [ ] Share release notes and user-facing documentation
- [ ] Thank team and stakeholders for their contributions

### Post-Release
- [ ] Send post-release summary with key metrics
- [ ] Share early adoption or usage data
- [ ] Communicate any known issues or follow-up work
- [ ] Solicit feedback from stakeholders
- [ ] Document lessons learned and improvements for next release

## Post-Project Phase

### Project Closure
- [ ] Distribute final project summary and outcomes
- [ ] Share retrospective insights and action items
- [ ] Confirm all deliverables are complete
- [ ] Archive project artifacts in accessible location
- [ ] Thank stakeholders and team for their contributions
- [ ] Conduct lessons learned session with key stakeholders

## Continuous Practices

### Ongoing Engagement
- [ ] Maintain open feedback channels
- [ ] Respond to stakeholder questions within agreed SLA
- [ ] Update stakeholders on unexpected changes promptly
- [ ] Celebrate wins and milestones
- [ ] Solicit input on process improvements

### Documentation Hygiene
- [ ] Keep project charter and status docs current
- [ ] Archive outdated information with clear timestamps
- [ ] Make project artifacts searchable and accessible
- [ ] Update stakeholder contact list as team changes occur

## Communication Best Practices

### Effective Updates
- **Be concise**: Stakeholders are busyâ€”lead with key information
- **Be honest**: Share challenges alongside successes
- **Be proactive**: Don't wait for stakeholders to ask for updates
- **Be visual**: Use charts, dashboards, or screenshots when helpful
- **Be consistent**: Maintain regular communication cadence

### Red Flags to Communicate Immediately
- Critical bugs or security issues
- Major timeline slippages (>20% delay)
- Scope creep that affects other teams
- Resource constraints threatening delivery
- Dependencies that are blocked or at risk
- Customer escalations or negative feedback
